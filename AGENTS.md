# AGENTS.md

Guidance for AI coding agents (Claude Code, Cursor, Aider, Copilot, etc.) and human contributors working on **bgpeek** — an open-source looking glass for ISPs and IX operators.

## Project

- **License:** Apache-2.0 · **Language:** English only in tracked files
- **Stack:** FastAPI + Pydantic v2, Jinja2 + HTMX + Tailwind (SSR, no SPA, no npm at runtime), PostgreSQL, Redis, Netmiko (SSH), Docker Compose
- **Auth:** API key, bcrypt, LDAP, OIDC (Keycloak), JWT cookie

For currently shipped features, supported platforms, and roadmap, see `README.md` and `CHANGELOG.md`.

## Layout

```
src/bgpeek/
  api/        FastAPI routers (REST + HTMX partials)
  core/       SSH, parsing, RPKI, rate limit, webhooks, commands
  db/         SQLAlchemy models, session, migration helpers
  models/     Pydantic schemas
  ui/         Jinja2 routes
  templates/  Server-rendered HTML
  static/     Committed Tailwind CSS + JS
migrations/   Alembic — schema only, no operator data
scripts/      Seed examples, admin helpers
tests/        pytest + pytest-asyncio
```

## Hard rules

- **No SPA / no npm runtime build.** Tailwind CSS is pre-built and committed. HTMX + server-rendered HTML is a deliberate choice.
- **No operator-specific data in tracked files.** Migrations ship `CREATE TABLE` only; seeds use documentation ASNs (RFC 5398) and well-known communities. Real ASNs, customer prefixes, or internal community taxonomies must be loaded via the admin API at deploy time.
- **Per-role output filtering is first-class.** Public users must not see more-specific prefixes (default: hide /25-/32 for v4, equivalent for v6). Do not regress this. Write denial-path tests when touching auth, rate limiting, or filtering.
- **No secrets in code.** All secrets via env vars; `.env.example` is the source of truth.
- **Type hints required.** Mypy must pass on `src/`.
- **No AI attribution in code or public docs.** Co-author trailers on commits are fine; no "generated by AI" phrasing in source, docstrings, or README.

## Workflow

```bash
make install    # uv sync --extra dev
make dev        # docker compose up (Postgres + Redis + app + tailwind watcher)
make migrate    # Alembic migrations
make css        # one-shot Tailwind CSS build (via Docker, no host binary)
make check      # lint + format-check + mypy + test (run before every PR)
```

In dev, a `tailwind` container watches `src/bgpeek/templates/**/*.html` and
rebuilds `static/css/tailwind.css` on every save. No host tailwindcss
binary is required. The compiled CSS is gitignored (it's a build
artifact); in production the Dockerfile builds it at image build time.

**Contributing a change (external contributors):**

1. Check open issues, milestones, and `CHANGELOG.md` first — avoid overlap with work in progress.
2. Fork → branch → one feature per commit, descriptive message.
3. Add or update tests. New features without tests will be asked to add them.
4. `make check` must pass locally and in CI.
5. Open a PR; the maintainer picks the merge strategy.
6. Do not push directly to `main` or tag releases — the maintainer handles both.

## Version bump checklist (bgpeek)

When changing the bgpeek release version (for example `1.3.1` -> `1.3.2`), update all of these together:

1. **Python package version:** `pyproject.toml` -> `[project].version`
2. **Runtime/app version constant:** `src/bgpeek/__init__.py` -> `__version__`
3. **Lockfile package entry:** `uv.lock` -> `[[package]] name = "bgpeek"` block `version` field (regenerate lockfile if needed)
4. **Changelog entry:** `CHANGELOG.md`
   - add new `## [x.y.z] - YYYY-MM-DD` section at the top
   - add/update the `[x.y.z]: https://github.com/xeonerix/bgpeek/releases/tag/vx.y.z` link at the bottom

Quick verification before commit:

```bash
rg -n '^version = "|__version__ = "|^## \\[[0-9]+\\.[0-9]+\\.[0-9]+\\]|^\\[[0-9]+\\.[0-9]+\\.[0-9]+\\]:' \
  pyproject.toml src/bgpeek/__init__.py CHANGELOG.md uv.lock
```

## Adding a new vendor platform

To add a platform (e.g. Nokia SR OS, MikroTik RouterOS), touch these four places:

1. **Command templates:** add entries to `_COMMAND_TABLE` in `src/bgpeek/core/commands.py` for each `(platform, QueryType, Family)` tuple — BGP_ROUTE, PING, TRACEROUTE × v4/v6.
2. **Netmiko driver:** use the Netmiko `device_type` string (e.g. `cisco_xr`, `arista_eos`). If the vendor needs a custom driver, wire it in `src/bgpeek/core/ssh.py`.
3. **BGP output parser:** if the platform's BGP output format isn't already handled, extend `src/bgpeek/core/bgp_parser.py`. Structured fields parsed per route include prefix, next-hop, AS path, communities, local-pref, MED, origin, and best-route flag.
4. **Tests:** add to `tests/test_commands.py` (command building) and `tests/test_bgp_parser.py` (parsing). Use documentation ASNs/IPs (RFC 5398, RFC 5737, RFC 3849) — avoid real customer prefixes.

Keep platform support generic — no vendor-specific hacks driven by one operator's config.

## Security

- **Do not file vulnerabilities as public issues.** See `SECURITY.md` for private disclosure.
- SSH credentials are Fernet-encrypted at rest. Do not log credentials or raw SSH transcripts.
- Webhook payloads are signed HMAC-SHA256; don't weaken this.
- Rate limits use a Redis sliding window (per-IP, per-user, per-API-key).
- Router banners are stripped from SSH output to avoid leaking MOTDs.

## Don'ts (quick reference)

- Don't convert the UI to a SPA or introduce a Node build step at runtime.
- Don't commit secrets, real ASNs of specific networks, customer prefixes, or internal community taxonomies.
- Don't batch unrelated changes in one commit.
- Don't rewrite unrelated code as part of a feature PR — open a separate issue.
- (External contributors) Don't push to `main` or tag releases.
